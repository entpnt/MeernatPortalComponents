# Git Commit

Stage and commit work following Meernat commit conventions.

## Commit Rules — Non-Negotiable

1. **No Anthropic signatures** — never include `Co-Authored-By: Claude` or any Anthropic attribution in any commit message. No exceptions.
2. **No AI attribution** — do not mention Claude, AI, or any assistant in commit messages.
3. **Commit frequently** — commit after each logical unit of work is complete. Do not accumulate large batches of unrelated changes in a single commit.
4. **One concern per commit** — each commit should do one thing. A migration, a component, a hook, a fix — not all of them together.
5. **No WIP commits to shared branches** — commits on feature branches must be complete and working. Never commit broken code.

## Commit Message Format

```
<type>(<scope>): <short description>

[optional body — what and why, not how]
```

**Types:**
- `feat` — new feature or capability
- `fix` — bug fix
- `refactor` — code change that neither fixes a bug nor adds a feature
- `chore` — migrations, config changes, dependency updates
- `test` — adding or updating tests
- `docs` — documentation only

**Scope** — the area of the codebase (optional but helpful):
- `auth`, `billing`, `network`, `installer`, `partner`, `db`, `api`, `ui`

**Examples:**
```
feat(billing): add payment method selector component
fix(auth): handle expired Clerk session on route guard
chore(db): add migration for network_device_assignments table
refactor(ui): extract shared StatusBadge from NetworkTable
test(billing): add unit tests for usePaymentMethods hook
```

**Short description rules:**
- Lowercase, no period at the end
- Present tense ("add" not "added")
- Under 72 characters
- Describes what the commit does, not what you did

## When to Commit

Commit after each of these checkpoints — don't wait until everything is done:

- A migration file is written and reviewed
- A new component is complete and rendering
- A hook or service is complete and tested
- A bug is fixed and verified
- A route is wired up
- Tests are written for a module
- A refactor of a single file or function is complete

**Rule of thumb:** if you can write a one-line commit message that accurately describes the change, it's the right size to commit.

## Workflow

### Step 1: Check what's changed

```bash
git status
git diff --stat
```

Review the changes. If multiple unrelated things are modified, stage them separately into multiple commits.

### Step 2: Stage selectively

Stage only what belongs in this commit:

```bash
# Stage specific files
git add src/components/PaymentSelector.tsx
git add src/components/PaymentSelector.test.tsx

# Or stage a directory
git add src/features/billing/

# Never use git add . without reviewing what it includes
```

### Step 3: Write the commit message

```bash
git commit -m "feat(billing): add PaymentSelector component with card/bank options"
```

For commits that need more context, use a body:

```bash
git commit -m "fix(network): correct RLS policy for jamestown_bpu subscriber reads

Previous policy was checking auth.uid() directly instead of the Clerk
JWT sub claim, causing all subscriber queries to return empty results
for network-isolated sessions."
```

### Step 4: Verify

```bash
git log --oneline -5
```

Confirm the commit appears correctly before continuing.

## What Never Goes in a Commit Message

- `Co-Authored-By: Claude ...`
- `Generated by Claude`
- `AI-assisted`
- `🤖`
- Any reference to the AI assistant that wrote the code

The commit history belongs to the engineering team. Keep it clean.

## Stacking Commits on a Feature Branch

For a typical feature, aim for this kind of history:

```
feat(db): add subscription_events table migration
feat(db): add get_subscription_history RPC function
feat(api): add subscription-history edge function
feat(crm): add useSubscriptionHistory hook
feat(crm): add SubscriptionHistoryTable component
feat(crm): wire SubscriptionHistory page to route
test(crm): add tests for SubscriptionHistoryTable
```

Seven small commits, each reviewable in isolation, each clearly describing what changed. This is far better than one large commit: `feat: add subscription history feature`.
